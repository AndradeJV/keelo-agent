# ISO/IEC 29119 - Padrões de Teste de Software

## Introdução

A ISO/IEC 29119 é uma família de padrões internacionais para teste de software, cobrindo conceitos, processos, documentação, técnicas e automação.

## Partes do Padrão

| Parte | Título | Descrição |
|-------|--------|-----------|
| 29119-1 | Conceitos e Definições | Vocabulário e conceitos de teste |
| 29119-2 | Processos de Teste | Processos organizacionais e de projeto |
| 29119-3 | Documentação de Teste | Templates de documentos |
| 29119-4 | Técnicas de Teste | Técnicas de design de teste |
| 29119-5 | Teste Baseado em Palavras-chave | Framework para automação |

## Parte 1: Conceitos e Definições

### Termos Fundamentais

| Termo | Definição |
|-------|-----------|
| **Teste** | Conjunto de atividades para avaliar qualidade e identificar defeitos |
| **Caso de Teste** | Conjunto de precondições, entradas, ações, resultados esperados |
| **Objeto de Teste** | Item a ser testado (sistema, componente, documento) |
| **Base de Teste** | Corpo de conhecimento usado para derivar casos de teste |
| **Oráculo de Teste** | Fonte para determinar resultado esperado |
| **Testware** | Artefatos produzidos durante teste |

### Modelo Conceitual

```
Base de Teste ──► Condições de Teste ──► Casos de Teste ──► Procedimentos de Teste
     │                    │                    │                      │
     ▼                    ▼                    ▼                      ▼
Requisitos          Itens a testar      Entradas/Saídas        Passos de execução
Especificações      Funcionalidades     Dados de teste         Scripts
Código              Riscos              Resultados esperados   Automação
```

## Parte 2: Processos de Teste

### Processos Organizacionais

#### Política de Teste
Define diretrizes de teste para a organização.

```yaml
Política de Teste:
  objetivos:
    - Encontrar defeitos antes da produção
    - Garantir conformidade com requisitos
    - Reduzir custos de correção
  
  princípios:
    - Teste baseado em risco
    - Automação onde viável
    - Melhoria contínua
  
  padrões:
    - Cobertura mínima de código: 80%
    - Testes automatizados em CI/CD
    - Revisão de código obrigatória
```

#### Estratégia de Teste
Define abordagem de teste para projeto ou produto.

```yaml
Estratégia de Teste - Projeto XYZ:
  níveis:
    unitário:
      responsável: Desenvolvedores
      cobertura: 80% branches
      ferramentas: Vitest
    
    integração:
      responsável: QA + Dev
      foco: APIs, banco de dados
      ferramentas: Supertest
    
    sistema:
      responsável: QA
      foco: E2E, performance
      ferramentas: Playwright, k6
    
    aceite:
      responsável: PO + Usuários
      foco: Critérios de negócio
      formato: UAT manual
  
  automação:
    prioridade: 
      - Smoke tests (crítico)
      - Regressão funcional
      - Performance baseline
    
  riscos:
    - Prazo apertado: Priorizar testes de risco alto
    - Equipe nova: Investir em treinamento
```

### Processos de Gerenciamento de Teste

#### Planejamento de Teste

```yaml
Plano de Teste - Sprint 15:
  escopo:
    incluído:
      - Nova funcionalidade de checkout
      - Correções de bugs P1
    excluído:
      - Módulo de relatórios (próxima sprint)
  
  abordagem:
    - Testes funcionais: Todos os critérios de aceite
    - Testes de regressão: Suite automatizada
    - Testes exploratórios: 4 horas
  
  cronograma:
    análise: 2 dias
    design: 2 dias
    execução: 3 dias
    report: 1 dia
  
  recursos:
    - QA: 2 pessoas
    - Ambiente: Staging
    - Dados: Cópia anonimizada de produção
  
  critérios_saída:
    - 100% casos críticos passando
    - 0 defeitos bloqueadores
    - 95% cobertura de requisitos
```

#### Monitoramento e Controle

```yaml
Métricas de Teste - Sprint 15:
  progresso:
    casos_planejados: 150
    casos_executados: 120 (80%)
    casos_passando: 115 (96%)
    casos_falhando: 5 (4%)
  
  defeitos:
    encontrados: 12
    corrigidos: 8
    pendentes: 4
    severidade:
      crítico: 0
      alto: 2
      médio: 6
      baixo: 4
  
  cobertura:
    requisitos: 92%
    código: 78%
  
  tendência:
    - Defeitos encontrados diminuindo (curva estabilizando)
    - Cobertura aumentando conforme esperado
```

## Parte 3: Documentação de Teste

### Documentos de Gerenciamento

#### Política de Teste Organizacional
```markdown
# Política de Teste - [Organização]

## 1. Objetivo
Estabelecer diretrizes para atividades de teste.

## 2. Escopo
Aplicável a todos os projetos de software.

## 3. Princípios
- Todo código deve ser testado antes de produção
- Automação deve ser priorizada onde economicamente viável
- Testes devem ser rastreáveis aos requisitos

## 4. Responsabilidades
- Desenvolvedores: Testes unitários
- QA: Testes de sistema e aceite
- DevOps: Testes de performance e infraestrutura

## 5. Métricas
- Cobertura mínima: 80%
- Defeitos em produção: < 2 por release
```

#### Plano de Teste
```markdown
# Plano de Teste - [Projeto] v1.0

## 1. Identificação
- Projeto: Sistema de E-commerce v2.0
- Versão do Plano: 1.0
- Data: 2024-01-15
- Autor: QA Lead

## 2. Escopo
### 2.1 Itens de Teste
- Módulo de Checkout
- Gateway de Pagamento
- Notificações por Email

### 2.2 Fora do Escopo
- Módulo de Relatórios (teste separado)

## 3. Abordagem
- Níveis: Integração, Sistema, Aceite
- Tipos: Funcional, Performance, Segurança
- Técnicas: EP, BVA, Exploratório

## 4. Critérios
### 4.1 Entrada
- Build estável disponível
- Ambiente de teste configurado
- Dados de teste preparados

### 4.2 Saída
- 100% casos críticos passando
- 0 defeitos bloqueadores abertos
- Stakeholders aprovaram

## 5. Cronograma
| Fase | Início | Fim |
|------|--------|-----|
| Design | 15/01 | 19/01 |
| Execução | 22/01 | 02/02 |
| Report | 05/02 | 06/02 |

## 6. Riscos
| Risco | Mitigação |
|-------|-----------|
| Atraso na entrega | Buffer de 2 dias |
| Ambiente instável | Backup local |
```

### Documentos de Design

#### Especificação de Caso de Teste
```markdown
# Caso de Teste: TC-CHECKOUT-001

## Identificação
- ID: TC-CHECKOUT-001
- Título: Checkout com cartão de crédito válido
- Prioridade: Alta
- Tipo: Funcional
- Nível: Sistema

## Rastreabilidade
- Requisito: REQ-PAY-001
- História: US-45

## Precondições
- Usuário logado
- Carrinho com pelo menos 1 item
- Endereço de entrega cadastrado

## Dados de Teste
| Campo | Valor |
|-------|-------|
| Número cartão | 4111111111111111 |
| Validade | 12/25 |
| CVV | 123 |
| Nome | TESTE USER |

## Passos
| # | Ação | Resultado Esperado |
|---|------|-------------------|
| 1 | Clicar em "Finalizar Compra" | Tela de checkout exibida |
| 2 | Selecionar "Cartão de Crédito" | Form de cartão exibido |
| 3 | Preencher dados do cartão | Campos validados |
| 4 | Clicar em "Pagar" | Loading exibido |
| 5 | Aguardar processamento | Tela de confirmação |

## Resultado Esperado Final
- Pedido criado com status "Pago"
- Email de confirmação enviado
- Estoque atualizado

## Pós-condições
- Pedido visível em "Meus Pedidos"
- Pagamento registrado no gateway

## Notas
- Usar sandbox do gateway em homologação
```

## Parte 4: Técnicas de Teste

### Catálogo de Técnicas

| Categoria | Técnicas |
|-----------|----------|
| **Specification-based** | Equivalence Partitioning, Boundary Value Analysis, Decision Table, State Transition, Use Case Testing |
| **Structure-based** | Statement Coverage, Decision Coverage, Condition Coverage, Path Coverage |
| **Experience-based** | Error Guessing, Exploratory Testing, Checklist-based |

### Seleção de Técnicas

```yaml
Critérios de Seleção:
  tipo_sistema:
    - Crítico (saúde, financeiro): Tabela de decisão, MC/DC
    - Web comercial: EP, BVA, Exploratório
    - Embedded: Transição de estado, Cobertura
  
  base_teste:
    - Requisitos claros: Técnicas baseadas em especificação
    - Código disponível: Técnicas estruturais
    - Pouca documentação: Técnicas baseadas em experiência
  
  risco:
    - Alto: Múltiplas técnicas combinadas
    - Médio: Uma técnica sistemática + exploratório
    - Baixo: Checklist ou smoke test
```

## Parte 5: Teste Baseado em Palavras-chave

### Estrutura

```yaml
# Keywords são ações reutilizáveis
keywords:
  Login:
    parameters: [username, password]
    steps:
      - navigate: /login
      - fill: username, ${username}
      - fill: password, ${password}
      - click: submit
      - verify: dashboard visible

  AddToCart:
    parameters: [productId]
    steps:
      - navigate: /products/${productId}
      - click: add-to-cart
      - verify: cart count increased

# Test cases usam keywords
test_cases:
  - name: Complete Purchase
    steps:
      - Login: [user@test.com, password123]
      - AddToCart: [PROD-001]
      - AddToCart: [PROD-002]
      - Checkout: [credit_card]
      - VerifyOrderConfirmation: []
```

### Benefícios

- Reutilização de componentes de teste
- Abstração do código de automação
- Fácil manutenção
- Legível por não-técnicos
