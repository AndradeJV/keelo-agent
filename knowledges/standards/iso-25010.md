# ISO/IEC 25010 - Modelo de Qualidade de Produto

## Introdução

A ISO/IEC 25010:2011 define um modelo de qualidade para produtos de software, identificando 8 características de qualidade e suas sub-características. É parte da família SQuaRE (Systems and Software Quality Requirements and Evaluation).

## 1. Adequação Funcional (Functional Suitability)

**Definição**: Grau em que o produto fornece funções que atendem às necessidades declaradas e implícitas.

### Sub-características

| Sub-característica | Definição | Cenário de Teste |
|--------------------|-----------|------------------|
| **Completude** | Grau em que as funções cobrem todas as tarefas e objetivos | Todas as funcionalidades do requisito estão implementadas |
| **Correção** | Grau em que os resultados estão corretos e precisos | Cálculos retornam valores corretos |
| **Apropriação** | Grau em que as funções facilitam as tarefas | Fluxo permite completar tarefa eficientemente |

### Exemplos de Teste

```gherkin
# Completude
Cenário: Sistema oferece todas as operações CRUD
  Dado que sou usuário administrador
  Quando acesso o módulo de produtos
  Então devo poder criar produto
  E devo poder visualizar produto
  E devo poder editar produto
  E devo poder excluir produto

# Correção
Cenário: Cálculo de impostos está correto
  Dado produto com valor R$ 100
  E alíquota ICMS de 18%
  Quando calculo o imposto
  Então o valor deve ser R$ 18,00

# Apropriação
Cenário: Checkout em poucos passos
  Dado que tenho itens no carrinho
  Quando inicio o checkout
  Então devo completar a compra em no máximo 4 etapas
```

## 2. Eficiência de Desempenho (Performance Efficiency)

**Definição**: Desempenho relativo à quantidade de recursos usados sob condições estabelecidas.

### Sub-características

| Sub-característica | Definição | Métrica Exemplo |
|--------------------|-----------|-----------------|
| **Tempo** | Tempo de resposta, processamento e throughput | Resposta < 500ms para 95% das requisições |
| **Recursos** | Quantidade de recursos utilizados | Uso de memória < 512MB |
| **Capacidade** | Limites máximos suportados | Suportar 1000 usuários simultâneos |

### Exemplos de Teste

```gherkin
# Tempo
Cenário: Página inicial carrega rapidamente
  Dado que acesso a página inicial
  Quando a página carrega completamente
  Então o tempo deve ser menor que 2 segundos
  E o LCP (Largest Contentful Paint) deve ser menor que 2.5s

# Recursos
Cenário: Aplicação não consome memória excessiva
  Dado que uso o sistema por 8 horas
  Quando monitoro uso de memória
  Então não deve aumentar mais que 10% (memory leak)

# Capacidade
Cenário: Sistema suporta carga esperada
  Dado que 500 usuários acessam simultaneamente
  Quando executam operações típicas
  Então o tempo de resposta médio deve ser < 1 segundo
  E não deve haver erros de timeout
```

## 3. Compatibilidade (Compatibility)

**Definição**: Grau em que o produto pode trocar informações e/ou executar suas funções enquanto compartilha ambiente.

### Sub-características

| Sub-característica | Definição | Exemplo |
|--------------------|-----------|---------|
| **Coexistência** | Operar eficientemente com outros produtos | Sistema funciona com antivírus ativo |
| **Interoperabilidade** | Trocar informações com outros sistemas | Integração com API de pagamento |

### Exemplos de Teste

```gherkin
# Coexistência
Cenário: Sistema funciona com outras aplicações
  Dado que tenho browser com múltiplas abas abertas
  E outras aplicações consumindo recursos
  Quando uso o sistema
  Então deve funcionar normalmente sem degradação severa

# Interoperabilidade
Cenário: Integração com gateway de pagamento
  Dado que inicio pagamento via Stripe
  Quando o pagamento é processado
  Então a confirmação deve ser recebida
  E o status do pedido deve atualizar automaticamente
```

## 4. Usabilidade (Usability)

**Definição**: Grau em que o produto pode ser usado por usuários específicos para atingir objetivos com eficácia, eficiência e satisfação.

### Sub-características

| Sub-característica | Definição | Métrica |
|--------------------|-----------|---------|
| **Reconhecibilidade** | Reconhecer se produto é apropriado | Usuário entende propósito em < 10s |
| **Aprendizibilidade** | Facilidade para aprender | Completar tarefa básica sem ajuda |
| **Operabilidade** | Facilidade de operação e controle | Concluir tarefa em poucos passos |
| **Proteção contra erro** | Proteção contra erros do usuário | Confirmações em ações destrutivas |
| **Estética de interface** | Interface agradável | Score de satisfação visual > 4/5 |
| **Acessibilidade** | Uso por pessoas com limitações | Conformidade WCAG AA |

### Exemplos de Teste

```gherkin
# Aprendizibilidade
Cenário: Novo usuário completa cadastro sem ajuda
  Dado que sou novo usuário
  Quando tento fazer cadastro pela primeira vez
  Então devo completar sem consultar documentação
  E o tempo deve ser menor que 3 minutos

# Proteção contra erro
Cenário: Confirmação antes de excluir
  Dado que seleciono item para excluir
  Quando clico em excluir
  Então devo ver modal de confirmação
  E a ação deve requerer confirmação explícita

# Acessibilidade
Cenário: Navegação por teclado
  Dado que navego apenas com teclado
  Quando percorro todos os elementos interativos
  Então todos devem ser acessíveis via Tab
  E o foco deve ser visível
```

## 5. Confiabilidade (Reliability)

**Definição**: Grau em que o produto executa funções especificadas sob condições especificadas por período determinado.

### Sub-características

| Sub-característica | Definição | Métrica |
|--------------------|-----------|---------|
| **Maturidade** | Atende necessidades de confiabilidade | MTBF > 720 horas |
| **Disponibilidade** | Operacional quando necessário | Uptime > 99.9% |
| **Tolerância a falhas** | Opera apesar de falhas | Sistema continua com componente falho |
| **Recuperabilidade** | Recuperar após falha | Restore < 30 minutos |

### Exemplos de Teste

```gherkin
# Disponibilidade
Cenário: SLA de disponibilidade
  Dado que monitoro o sistema por 30 dias
  Quando verifico o uptime
  Então deve ser maior que 99.9%
  E downtime total deve ser menor que 43 minutos

# Tolerância a falhas
Cenário: Sistema continua com cache indisponível
  Dado que o Redis está fora do ar
  Quando acesso o sistema
  Então deve funcionar (mais lento)
  E devo ver alerta sobre degradação

# Recuperabilidade
Cenário: Restore de backup funciona
  Dado que tenho backup de ontem
  Quando executo procedimento de restore
  Então o sistema deve voltar a funcionar
  E todos os dados devem estar corretos
```

## 6. Segurança (Security)

**Definição**: Grau em que o produto protege informações e dados de forma que pessoas ou sistemas tenham acesso apropriado.

### Sub-características

| Sub-característica | Definição | Exemplo |
|--------------------|-----------|---------|
| **Confidencialidade** | Dados acessíveis apenas a autorizados | Criptografia de dados sensíveis |
| **Integridade** | Prevenção de modificação não autorizada | Hash de verificação |
| **Não-repúdio** | Ações podem ser provadas | Log de auditoria |
| **Responsabilização** | Ações rastreáveis ao responsável | Identificação de usuário em logs |
| **Autenticidade** | Identidade verificável | Autenticação multi-fator |

### Exemplos de Teste

```gherkin
# Confidencialidade
Cenário: Dados sensíveis são criptografados
  Dado que acesso o banco de dados diretamente
  Quando leio campo de senha
  Então deve estar em formato hash (não plaintext)
  E senhas diferentes devem ter hashes diferentes

# Integridade
Cenário: Tentativa de SQL Injection falha
  Dado que informo "'; DROP TABLE users; --" no campo nome
  Quando submeto o formulário
  Então a entrada deve ser sanitizada
  E não deve haver erro de SQL

# Autenticidade
Cenário: Login requer autenticação forte
  Dado que informo usuário e senha corretos
  Quando tento acessar área administrativa
  Então devo informar código MFA
  E sem MFA não devo ter acesso
```

## 7. Manutenibilidade (Maintainability)

**Definição**: Grau de eficácia e eficiência com que o produto pode ser modificado.

### Sub-características

| Sub-característica | Definição | Métrica |
|--------------------|-----------|---------|
| **Modularidade** | Componentes independentes | Mudança afeta < 3 módulos |
| **Reusabilidade** | Ativos podem ser usados em outros sistemas | Componentes publicados como lib |
| **Analisabilidade** | Facilidade para diagnosticar | Time to diagnose < 30min |
| **Modificabilidade** | Facilidade para modificar | Change effort < estimate |
| **Testabilidade** | Facilidade para testar | Cobertura > 80% |

### Exemplos de Teste

```gherkin
# Analisabilidade
Cenário: Logs facilitam diagnóstico
  Dado que ocorre erro em produção
  Quando analiso os logs
  Então devo encontrar stacktrace completo
  E devo identificar a causa em < 30 minutos

# Testabilidade
Cenário: Código tem cobertura adequada
  Dado que executo suite de testes
  Quando verifico relatório de cobertura
  Então cobertura de branches deve ser > 80%
  E funções críticas devem ter 100%
```

## 8. Portabilidade (Portability)

**Definição**: Grau de eficácia e eficiência com que o produto pode ser transferido de um ambiente para outro.

### Sub-características

| Sub-característica | Definição | Exemplo |
|--------------------|-----------|---------|
| **Adaptabilidade** | Adaptar a diferentes ambientes | Funciona em Linux e Windows |
| **Instalabilidade** | Facilidade de instalação | Install script < 5 minutos |
| **Substituibilidade** | Substituir outro produto | Migração de dados suportada |

### Exemplos de Teste

```gherkin
# Adaptabilidade
Cenário: Funciona em diferentes browsers
  Dado que acesso em Chrome, Firefox e Safari
  Quando uso as funcionalidades principais
  Então todas devem funcionar corretamente em todos

# Instalabilidade
Cenário: Deploy automatizado funciona
  Dado que tenho ambiente limpo
  Quando executo script de deploy
  Então o sistema deve estar funcional
  E o tempo total deve ser < 15 minutos
```
